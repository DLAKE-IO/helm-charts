# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Overview

This is a Helm charts repository that hosts multiple application charts. Charts are automatically published to GitHub Pages via chart-releaser when changes are merged to main.

## Available Charts

- **bookstack** - Self-hosted platform for organizing and storing information
- **cryptgeon** - Secure note sharing application with Redis backend
- **ocsinventory** - IT assets management and deployment solution with MariaDB backend
- **pingvin-share** - File sharing application (note: limited to 1 replica due to architecture)

## Development Commands

### Linting and Testing

```bash
# Lint charts (uses ct lint)
ct lint --config ./.github/configs/ct.yaml --lint-conf ./.github/configs/lintconf.yaml

# Install and test charts (requires kind cluster)
ct install --config ./.github/configs/ct.yaml

# List changed charts
ct --config ./.github/configs/ct.yaml list-changed
```

### Helm Operations

```bash
# Add required dependency repositories
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo add valkey https://valkey.io/valkey-helm/

# Update dependencies for a chart
helm dependency update charts/<chart-name>

# Template a chart to preview manifests
helm template <release-name> charts/<chart-name> -f charts/<chart-name>/values.yaml

# Package a chart
helm package charts/<chart-name>
```

### Documentation

All charts use helm-docs for automatic README generation. After making changes to Chart.yaml or values.yaml:

```bash
# Generate/update README.md for charts
helm-docs
```

## Chart Structure

Each chart follows the standard Helm structure:
- `Chart.yaml` - Chart metadata (version, appVersion, dependencies, maintainers)
- `values.yaml` - Default configuration values
- `templates/` - Kubernetes manifest templates
- `requirements.yaml` - Legacy dependency specification (only in older charts like bookstack)

Charts may include external dependencies:
- **cryptgeon**: Bitnami Redis (17.x.x)
- **ocsinventory**: Bitnami MariaDB (20.x.x)
- **pingvin-share**: Bitnami Common (2.x.x)

## CI/CD Workflows

The repository uses GitHub Actions for automation:

- **lint-test.yaml** - Runs on PRs to lint and test changed charts
- **helm-test.yml** - Runs on push/PR to main branch for CI testing
- **release.yaml** - Automatically packages and publishes charts to GitHub Pages on main branch pushes

## Chart Testing Configuration

Chart testing is configured via `.github/configs/ct.yaml`:
- Target branch: `main`
- Chart directory: `charts/`
- Validates maintainers and YAML
- Excludes deprecated charts

## Mandatory Rules for Chart Modifications

**These rules are non-negotiable. Every chart modification — no matter how small — must trigger all four steps below before the work is considered complete.**

### 1. Bump `version` in `Chart.yaml`

Every change to a chart's templates, values, helpers, or dependencies requires a version increment in `charts/<chart-name>/Chart.yaml`. Use Semantic Versioning to determine the correct level:

| Bump | When to use |
|------|-------------|
| **MAJOR** (`x+1.0.0`) | Breaking changes: renamed/removed values keys, changed default behaviour that requires user action, incompatible dependency upgrades |
| **MINOR** (`x.y+1.0`) | New features: new values keys, new optional capabilities, new sub-chart integrations, new template resources |
| **PATCH** (`x.y.z+1`) | Bug fixes, documentation-only changes, non-breaking default tweaks, internal refactors with no user-visible effect |

Also increment `appVersion` in `Chart.yaml` when the upstream application version changes.

### 2. Update the chart `README.md`

After modifying `Chart.yaml`, `values.yaml`, or any template, update `charts/<chart-name>/README.md` to reflect the change. This file is generated by helm-docs; if helm-docs is available run it — otherwise update the file manually:

```bash
helm-docs   # regenerates all charts/*/README.md from README.md.gotmpl
```

The README must stay accurate for:
- New or removed values (add/remove rows in the Values table)
- Changed defaults (update the Default column)
- New dependencies (update the Requirements table)
- Version and AppVersion badges (update automatically via helm-docs or manually)
- Behavioural changes that affect operators (add a prose note if needed)

### 3. Update the root `README.md`

Update `/README.md` whenever a chart change is user-visible. This includes:

- Version badge for the chart in the **Chart Documentation** section (`![Version: X.Y.Z]`)
- Version cell in the **Available Charts** table
- Any installation example, feature description, or note that is now outdated

If the change is purely internal (template refactor, CI tweak) with no impact on how operators deploy or configure the chart, the root README does not need to change.

### 4. Add an entry to `CHANGELOG.md`

Every chart change must be recorded in `/CHANGELOG.md` in two places:

1. **`[Unreleased]` section** — add a bullet under the appropriate heading (`Added`, `Changed`, `Fixed`, `Removed`, or `Breaking`). Format:
   ```
   - **<chart-name>** version bump `OLD` → `NEW`; <concise description of change>
   ```

2. **Versioned chart section** (e.g. `## bookstack`) — prepend a new `### [X.Y.Z] — YYYY-MM-DD` block with one bullet per logical change. Example:
   ```markdown
   ### [2.4.0] — 2026-03-06
   - Added `valkey.auth.existingSecret` value; deployment now injects `REDIS_PASSWORD` from the referenced Secret
   ```

Use today's date. Keep descriptions factual and operator-focused (what changed and why it matters, not how the template was edited).

## Common Patterns

Charts in this repository commonly include:
- Ingress support with configurable annotations and className
- Persistent volume claims for stateful applications
- Security contexts with fsGroup and runAsUser/runAsNonRoot
- ServiceAccount creation with configurable annotations
- External database support (disable internal DB, provide external connection details)
- Health probes (liveness/readiness) where applicable
- Resource limits/requests (often left unspecified by default)

## Release Process

Charts are released automatically when merged to main:
1. chart-releaser packages changed charts
2. Creates GitHub releases with chart archives
3. Updates the Helm repository index on GitHub Pages

Users can add the repository with:
```bash
helm repo add dlake https://dlake-io.github.io/helm-charts/
```
